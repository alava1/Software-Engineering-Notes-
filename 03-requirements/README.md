# Requirements Engineering

## Overview

Requirements Engineering is the process of identifying, documenting, analyzing, validating, and managing the needs and expectations of stakeholders for a software system. It is one of the most important phases of software engineering because it lays the foundation for successful software development.

Poorly defined requirements often lead to project delays, increased costs, and software that does not meet user expectations.

---

# Objectives

The objectives of Requirements Engineering are to:

- Understand stakeholder needs.
- Define clear and complete software requirements.
- Reduce misunderstandings between developers and stakeholders.
- Minimize project risks.
- Ensure the final software satisfies business and user needs.

---

# Types of Requirements

## 1. Functional Requirements

Functional requirements describe what the software should do.

Examples:

- Users can register an account.
- Users can log in securely.
- Administrators can manage users.
- Customers can place orders.
- Reports can be generated.

---

## 2. Non-Functional Requirements

Non-functional requirements describe how the software should perform.

Examples:

- Fast response time.
- High availability.
- Data security.
- Easy to use.
- Scalability.
- Reliability.
- Maintainability.

---

# Requirements Gathering Techniques

Common techniques include:

- Interviews
- Questionnaires
- Observation
- Brainstorming
- Workshops
- Document Analysis
- Prototyping

Each technique helps gather accurate information from stakeholders.

---

# Stakeholders

Stakeholders are individuals or organizations affected by the software project.

Examples:

- Customers
- End Users
- Project Managers
- Developers
- Testers
- Business Owners

---

# User Stories

A User Story is a simple description of a feature from the user's perspective.

Format:

> As a **<user>**, I want **<feature>** so that **<benefit>**.

Example:

> As a student, I want to view my grades online so that I can track my academic progress.

---

# Use Cases

A Use Case describes how a user interacts with a system to achieve a specific goal.

Example:

### Use Case: User Login

**Actor**
- Registered User

**Precondition**
- User has an existing account.

**Steps**
1. Open the login page.
2. Enter username and password.
3. Click Login.
4. System verifies credentials.
5. User is redirected to the dashboard.

**Postcondition**
- User successfully accesses the system.

---

# Software Requirements Specification (SRS)

An SRS is a document that defines all software requirements.

Typical sections include:

- Introduction
- Purpose
- Scope
- Functional Requirements
- Non-functional Requirements
- System Features
- Constraints
- Assumptions

The SRS acts as an agreement between stakeholders and the development team.

---

# Challenges in Requirements Engineering

Common challenges include:

- Changing requirements
- Poor communication
- Incomplete information
- Unrealistic expectations
- Conflicting stakeholder needs

---

# Best Practices

- Engage stakeholders early.
- Ask clear questions.
- Document requirements thoroughly.
- Validate requirements with stakeholders.
- Keep requirements updated.
- Prioritize requirements based on business value.

---

# Summary

Requirements Engineering ensures that software is built according to stakeholder needs. Clear, complete, and validated requirements help reduce risks, improve software quality, and increase the chances of project success.
